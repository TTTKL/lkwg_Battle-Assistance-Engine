# 项目阶段总结

更新日期：2026-04-12

## 当前结论

项目目前仍然是两条主线并行：

- `engine_core/`（兼容别名：`battle_analyzer/`、`new_one/`）
  负责新的完整信息分析引擎，以及事件驱动的 `agent_runtime` 原型。
- `legacy_workspace/Calculator/`（兼容历史路径：`calculator_workspace/Calculator/`、`20260402114626/Calculator/`）
  负责当前主要可用的人机交互入口，也是最近迭代最频繁的工作台。

当前阶段已经不是“能跑通的原型”，而是“可以持续验证和修正的对战分析工作台”：

- 旧网页已能完成编队导入、首发选择、引擎态回合执行、明箱/暗箱分析。
- 右侧引擎分析区可显示推荐动作、相对评分、两位小数胜率、搜索统计。
- `engine_core/agent_runtime` 已具备导入、回滚、清空、批次查询等运行时能力。

## 已完成工作

### 1. `engine_core/` 引擎与运行时框架

- 完整信息分析主链已成型：
  - 状态模型
  - 行动生成
  - 战斗推进
  - 局面评估
  - 搜索
- `agent_runtime` 已接入事件式运行时能力：
  - 导入敌方血量/能量快照
  - 清空事件信息
  - 导入批次查询
  - 导入批次回滚
  - CLI / API / Web runtime console
- 状态底盘已做第一轮加固：
  - `PetInstance.runtime_flags`
  - 兼容旧的 `pet._xxx`
  - `copy()` 带上运行时状态
  - 一批高风险 `_xxx` 访问已改为 helper

### 2. 旧网页工作台 `Calculator/`

- 首页主流程已切换为暗箱优先，明箱保留为后置模式。
- **AI 引擎已接入 Python API**：前端 JS 引擎代码已彻底删除，所有搜索/评估/推荐均通过 `fetch` 调用 `/api/analyze`（同域，由 `server.py` 统一提供）。
- **一键启动**：`python server.py` 同时提供静态文件和 Python 引擎 API（端口 5173）。
- 默认编队机制已固定：
  - `默认编队1`：我方默认
  - `默认编队2`：敌方默认，仅明箱自动导入
- 默认编队删除保护已加上，`默认编队1 / 默认编队2` 不可删除。
- 战斗启动流程已拆成两阶段：
  - 导入双方队伍
  - 第 0 回合点击“准备首发”，分别选择双方首发后进入第 1 回合
- 首发确认后会立即触发引擎分析，不再等执行完第一回合才出推荐。
- 技能栏与战斗区交互已继续收口：
  - 选中技能 / 聚能 / 换宠都有明确高亮
  - 推荐技能改为卡片高亮，不再插入额外徽标挤占技能图标
  - 蓝色 `~数字` 预计伤害标记已移除
- 敌方技能观测链路已落地：
  - 已观测 / 预测标记
  - 点击敌方技能登记为已观测
  - 每只敌方宠独立保存证据
- 暗箱敌方个体估计已改为分布式，而不是固定真值假设。
- 已补一轮”页面状态 ↔ 引擎请求/返回”的同步收口：
  - 前端请求会携带当前技能顺序、实时能耗、冷却信息
  - 引擎状态格式化结果会返回当前技能列表，而不再只返回血量/能量
  - 技能栏展示、行动合法性、日志中的能耗读取已优先使用实时技能状态，避免被静态 `battle_data.json` 覆盖
  - 前端 `buildApiPayloadFromB()` 已增加 `player_team_state / opponent_team_state`，传递首领化/愿力冲击选择
  - `/api/analyze` 现在返回引擎处理后的完整 `state`、合法行动和 action panel，前端据此同步技能顺序、能耗、冷却与可选动作
  - `/api/resolve` 已接入网页端回合推进，执行后直接采用引擎返回的下一状态

### 3. 搜索与推荐逻辑修正

- 明箱搜索已改成“双方同回合同步选行动”的对称求值，而不是单边轮流近似。
- 同速同优先级时，搜索中保留 50/50 概率分支求期望，实战结算也仍保留随机先后手。
- 胜率显示已精确到小数点后两位。
- 胜率估算改为基于“我方搜索分数 vs 对方搜索分数”的相对优势，降低系统性基线偏移。
- 搜索参数已上调：
  - 更深搜索层数
  - 更大节点预算
  - 更长时间预算
  - 更宽候选动作保留数

### 4. 机制与合法性修正

- 聚能在满能量时仍然允许使用。
- 聚能现在按“状态类行动”处理，可被 `应对状态` 正常应对。
- 搜索和实战层都已同步聚能规则，避免推荐与执行不一致。
- 已修复“推荐换成当前在场精灵”这种非法换宠动作。
- 换宠推荐现在会明确显示换哪一只精灵。
- 已确认并保留槽位 / 传动 / 永久改耗主干实现；`test_slot_effects.py` 当前通过。

## 当前主要风险

### 1. 旧网页仍然承载了过多主流程逻辑

- ~~搜索推荐与 UI 渲染仍然耦合较深~~ （已解耦：搜索走 Python API，UI 仅渲染结果）
- 暗箱模式的技能预测和 IV 估计仍在前端 JS 中处理
- 前端默认已切到引擎权威状态；旧本地回合执行代码仍保留兼容壳，但在 `ENGINE_AUTHORITY=true` 且 `LEGACY_LOCAL_BATTLE_ENABLED=false` 时会被直接短路
- 明箱 / 暗箱共用逻辑较多，回归时要格外注意互相影响

### 2. 搜索结果仍未完全等同于“稳定最优策略”

- 当前更接近工程上可用的同步行动搜索，而不是严格混合策略解
- 深度、截断和快速排序仍会影响推荐稳定性
- 明箱胜率偏差已大幅收口，但仍需要继续做镜像局面校准

### 3. `engine_core/` 底盘仍需继续加固

- 仍有旧式 `_xxx` 状态残留
- `trait_processor` 体量仍偏大
- `copy()` / replay 保护只做了第一轮
